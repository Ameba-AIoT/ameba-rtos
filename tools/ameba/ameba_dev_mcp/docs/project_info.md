# `project_info.json5` 配置说明

`<SDK_ROOT>/project_info.json5` 描述每个 SoC 的烧录布局（image 文件 + flash 地址）。
MCP 的 `flash_firmware_tool(alias)` 在烧录前会按本文件解析镜像列表。

> 模板：[`project_info.template.json5`](project_info.template.json5) — 仅在你想"跳过
> build 直接手填 manual 模式"时拿来打个底；正常流程下不需要。
>
> 本文件**不需要**手动从零创建：只要执行过一次 `build_firmware`，MCP 会自动生成 / 增量更新它。

---

## 1. 顶层结构

```json5
/*
 * project_info.json5 — flashing-image config for each SoC.
 */
{
  "schema_version": 1,

  /* Global defaults; each project may override. */
  "defaults": {
    "memory_type": "nor",
    "chip_erase": false
  },

  /* Keyed by SoC name (matches build_<SOC>/). */
  "projects": {
    "RTL8721F": { ... },
    "RTL8730E": { ... }
  }
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `schema_version` | int | 当前为 `1`；后续不兼容升级会递增 |
| `defaults.memory_type` | `"nor"` / `"nand"` / `"ram"` | 默认 flash 类型 |
| `defaults.chip_erase` | bool | 默认是否全片擦除 |
| `projects` | dict | key 为 SoC 名（与 `build_<SOC>/` 对应） |

---

## 2. 两种模式：`auto` / `manual`

每个 project 必须显式声明 `flash_layout_setting_mode`：

| 模式 | 谁拥有数据 | MCP 行为 |
|---|---|---|
| `auto` | MCP（`build_firmware` 写入） | 编译成功后 **覆盖** `build_dir` 与 `images[]`；烧录前会重新实时解析 SDK 头并对账 |
| `manual` | 用户 | MCP 永远不修改这个 SoC 的条目，只做合法性校验 |

### 2.1 auto 模式示例

```json5
"RTL8721F": {
  "flash_layout_setting_mode": "auto",
  "build_dir": "/home/me/sdk/build_RTL8721F",
  "memory_type": "nor",

  /* === generated by build, do not edit === */
  "images": [
    { "type": "IMG_BOOT",      "path": "/home/me/sdk/build_RTL8721F/build/project_km4tz/image/km4tz_boot_all.bin",   "start_addr": "0x08000000", "end_addr": "0x08013FFF" },
    { "type": "IMG_APP_OTA1",  "path": "/home/me/sdk/build_RTL8721F/build/project_km4ns/image/km4ns_image2_all.bin", "start_addr": "0x08014000", "end_addr": "0x081FFFFF" }
  ]
}
```

> ⚠️ `/* === generated by build, do not edit === */` 区域 **不要手工改**。下一次 `build_firmware` 会再次覆盖。
> 如果你需要自定义某个 SoC 的镜像，把 `flash_layout_setting_mode` 改成 `manual`。

### 2.2 manual 模式示例

```json5
"RTL8730E": {
  "flash_layout_setting_mode": "manual",
  "build_dir": "/home/me/sdk/build_RTL8730E",
  "memory_type": "nor",
  "images": [
    {
      "path": "/home/me/sdk/build_RTL8730E/km0_boot_all.bin",
      "start_addr": "0x08000000",
      "end_addr": "0x0800FFFF"
    },
    {
      "path": "/home/me/sdk/build_RTL8730E/km4_image2_all.bin",
      "start_addr": "0x08010000"
      /* end_addr 缺省时按 filesize 推算，仅用于区段不重叠校验 */
    },
    {
      /* VFS 类镜像可能不存在；optional=true 时缺失只警告不阻断 */
      "path": "/home/me/sdk/build_RTL8730E/vfs.bin",
      "start_addr": "0x083E0000",
      "optional": true
    }
  ]
}
```

## 3. `ProjectEntry` 字段详表

| 字段 | 必填 | 类型 | auto | manual |
|---|---|---|---|---|
| `flash_layout_setting_mode` | ✓ | `"auto"` / `"manual"` | 必为 `"auto"` | 必为 `"manual"` |
| `build_dir` | ✓ | string（**绝对路径**） | build 写入 | 用户手填 |
| `memory_type` | – | `"nor"` / `"nand"` / `"ram"` | 可选，缺省继承 `defaults.memory_type` | 同 |
| `images` | ✓ | `ProjectImageEntry[]` | build 写入 | 用户手填，至少 1 项 |

### `ProjectImageEntry`

| 字段 | 必填 | 类型 | 说明 |
|---|---|---|---|
| `type` | – | string | 区域名（`IMG_BOOT` / `IMG_APP_OTA1` / `VFS1` 等）；auto 模式由解析器填写 |
| `path` | ✓ | string（**绝对路径**） | `.bin` 文件 |
| `start_addr` | ✓ | hex string | `^0x[0-9a-fA-F]{1,8}$` |
| `end_addr` | – | hex string | manual 模式可省，省略时按文件大小推算 |
| `optional` | – | bool | `true` 时文件缺失只警告不阻断（VFS 等） |

> 严格 schema：`extra="forbid"`。出现未知字段会被 Pydantic 拒绝。

---

## 4. 自动增量更新规则

`build_firmware` 编译成功后会调 `loader.update_project_for_soc(...)`：

| 状态 | 行为 |
|---|---|
| 该 SoC 不存在 | 新增条目，模式 `auto`，写入 `build_dir` 与 `images[]` |
| 该 SoC 已是 `auto` | 覆盖 `build_dir` 与 `images[]`（保留 `memory_type`） |
| 该 SoC 是 `manual` | **不动**，保留用户配置 |

返回值会出现在 `build_firmware` 的结果里：

```json
"project_info": { "updated": true, "soc": "RTL8721F", "mode": "auto", "image_count": 3 }
```

---

## 5. 烧录前的校验与对账

1. **schema 校验**：`validate_project_config(soc=...)` 跑全部字段类型 / 路径绝对 / 区段重叠等检查。
2. **auto 模式实时对账**（`flash.py:_resolve_images_for_flash`）：
   - 跑 `flashcfg_parser.parse_project(SDK_ROOT, soc)` 实时解析。
   - 把解析值与 json5 中 `images[]` 比 `(type, start_addr, end_addr)`（小写 hex）。
   - 不一致 → 拒绝烧录，返回 `FLASH_LAYOUT_OUT_OF_SYNC`，提示先 `build_firmware`。
   - json5 为空 → 用解析值直接烧 + info 提示。
3. **文件存在校验**：所有 mandatory image 必须存在。

> 设计动机：用户可能用 CLI（`./ameba.py build`）编译但用 MCP 烧录。CLI 不会同步 json5，对账避免了"用旧地址烧到新固件"导致变砖。

## 6. 注释规范

- **只用** `/* … */` 块注释。
- **不要用** 行尾 `//` 注释（json5 库支持但项目规范禁止）。
- 自动生成的文件 header 注释会被 `templates.render_project_info` 重写，不要在 header 里添加自定义内容。

## 7. 常见错误与排查

| 错误码 | 原因 | 排查 |
|---|---|---|
| `PROJECT_CONFIG_MISSING` | 文件不存在 | 跑一次 `build_firmware`，自动生成 |
| `PROJECT_CONFIG_PARSE_ERROR` | json5 语法错（行尾 `//`、尾随逗号、不平衡的 `{}`） | 见 §6 注释规范 |
| `PROJECT_CONFIG_INVALID` | 字段类型 / 缺失 / 多余 | 看 `field_path` 定位 |
| `PROJECT_NOT_CONFIGURED` | 文件中没该 SoC | 先 build 该 SoC，或手动在 `projects` 下加 manual 条目 |
| `PROJECT_PATH_NOT_ABSOLUTE` | `build_dir` / image `path` 是相对路径 | 改为绝对路径 |
| `BUILD_DIR_MISSING` | auto 模式下 build_dir 不存在 | 先 `set_target` + `build_firmware` |
| `AUTO_IMAGES_MISSING` | auto 模式 images 为空 | flash 时 MCP 会尝试 live 解析；建议先 build_firmware 持久化 |
| `MANUAL_IMAGES_EMPTY` | manual 模式 images 为空 | 至少填一个 image |
| `IMAGE_PATH_NOT_ABSOLUTE` | image path 是相对路径 | 改为绝对路径 |
| `IMAGE_FILE_NOT_FOUND` | bin 文件不存在 | 先 build；或加 `"optional": true`（仅 VFS 等） |
| `IMAGE_RANGE_OVERLAP` | 多个 image 的 `[start, end]` 区间重叠 | 调地址或 `end_addr` |
| `FLASH_CFG_PARSE_FAILED` | live 解析 `ameba_flashcfg.{c,h}` 失败 | 看错误信息；可能是 SDK 头文件不完整 |
| `FLASH_LAYOUT_OUT_OF_SYNC` | json5 中布局过期 | 跑一次 MCP `build_firmware` 同步 |
| `PROFILE_NOT_FOUND` | `Devices/Profiles/` 缺该 SoC 的 `.rdev` | 检查 SoC 名拼写；联系 SDK 维护者补 profile |

## 9. 与 `board_info.json5` 的关系

- `project_info.json5` 描述 **"烧什么、烧哪儿"**（image + 地址）。
- [`board_info.json5`](board_info.md) 描述 **"在哪块板上烧"**（串口、远程、速率）。
- `flash_firmware_tool(alias)` 通过 alias 找到 board → 取 `board.soc` → 在 `project_info.json5` 找对应 `ProjectEntry`。
- 同一个 SoC 可以对应多块板（不同 alias 共享同一份 `ProjectEntry`）。
